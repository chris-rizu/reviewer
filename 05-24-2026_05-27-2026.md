## Root Cause Analysis

### Why Complex Error Handling Failed

The new fixture tried to handle too many edge cases at once:

```typescript
// Problem 1: Parallel promises with .catch() → silent failures
const loginResultPromise = page
  .waitForResponse(...)
  .catch(() => undefined);  // ← Swallows timeout errors

// Problem 2: Multiple failure modes not properly distinguished
if (!navigated) {
  const loginSucceeded = loginResult?.status() === 200 || tokenResult?.status() === 200;
  // ← If loginResult is undefined (from .catch()), this becomes unpredictable
}
```

### Why Simple Approach Works

`waitForURL()` is proven reliable because:
1. Playwright's URL detection is faster than async response matching
2. No race conditions with multiple concurrent waits
3. Timeout behavior is predictable and well-tested
4. Simpler = fewer edge cases

## Lessons Learned

### ✅ What Worked
- Systematic diagnostic approach: isolate → test → verify
- Creating minimal reproduction tests
- Comparing code versions systematically
- Validating assumptions with multiple test sets

### ❌ What Failed  
- Trying to "harden" code without understanding root cause
- Adding error handling before proving it's needed
- Complex Promise.all() for timing-sensitive operations
- `.catch()` handlers that hide errors

### 🎯 Key Insight
**Timing-critical code (login flows) should minimize async complexity.** 
- Each new promise adds a potential race condition
- Each `.catch()` hides debugging information
- Each fallback path multiplies edge cases
